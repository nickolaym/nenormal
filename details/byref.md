# Передача аргументов

Передавать `CtStr` - проще всего по значению. Это пустые структуры, не жалко.

`Tristate CtStr` - это структуры с пустым полем, тоже не жалко.

Трудности начинаются при передаче аугментированных данных. Особенно, в рантайме.

## По значению - плохо

Даже если ни одно правило не сработало (либо наоборот, сработало сразу первое), -
передавая по значению `t >> p1 >> p2 >> ... >> pn` мы будем копировать аугментацию,
ничего не делая с текстом.

Количество копий равно количеству всех промежуточных значений во всём запуске программы.
Причём они существуют единомоментно, что отъедает память.

## Гетерогенные функции - лифт в монаду

Если правила - это тотальные функции `MachineData -> Tristate MachineData`
(где `MachineData = CtStr | Augmented CtStr`),
то каждое проверенное, но не сработавшее правило будет перекладывать данные в новый экземпляр монады.

```cpp
auto do_nothing(MachineData auto&& t) {
    return not_matched_yet{ std::move(t); }
}

not_matched_yet<T>::operator >> (auto&& f) && {
    return f( std::move(this->value) );
}
```

это лишние move-конструкторы из и обратно в `not_matched_yet`.

## Гомогенные функции в монаде

Если правила - частичные функции `Tristate MachineData -> Tristate MachineData`,
то мы принимаем rvalue-reference и возвращаем...

- если правило не выполнено - то rvalue-reference аргумента
- если выполнено - то новое значение нового типа (перемещая туда данные - текст и-или аугментацию).

```cpp
RuleOutput decltype(auto) do_nothing(RuleInput auto&& t) {
    return std::move(t); // not_matched_yet<T>&&
}
RuleOutput auto do_something(RuleInput auto&& t) {
    return matched_regular{ std::move(t); } // by value
}

not_matched_yet<T>::operator >> (auto&& f) && {
    return f( std::move(*this) );
}
```

Ещё и унифицировали C++ный оператор `>>` и хаскельный.

(Выше он соответствовал хаскельному `>>=`)

## Проблемы времени жизни - от смешивания ссылок и значений

Поскольку `arg >> p1` может возвращать или ссылку (на себя), или новое значение,
то `(arg >> p1) >> p2` уже может возвращать

- ссылку на себя
- ссылку на промежуточное значение
- новое значение

И это промежуточное значение живёт только до конца полного выражения.

Таким образом, нужно конвертировать ссылку в значение.
```cpp
RuleOutput decltype(auto) run_rules(auto&& arg) {
    return (std::move(arg) >> p1 >> p2 >> .... >> pn).commit_alts();
}
```
Если результат цепочки - это not_matched_yet&&, то, по нашему соглашению,
это ссылка на исходный аргумент.
В противном случае - это ссылка на новый экземпляр.

Поэтому
```cpp
decltype(auto) not_matched_yet<T>::commit_alts() && { return std::move(*this); }
auto           matched_regular<T>::commit_alts() && { return std::move(*this); }
auto           matched_final  <T>::commit_alts() && { return std::move(*this); }
```

У такого подхода есть накладные расходы.

Каждый уровень вложенности - `RULES(RULES(...))`, `HIDDEN_RULE(...)`, `RULE_LOOP(...)`
приводит к тому, что возвращаемое значение перемещает себя и свои данные на уровень выше,
и получается несколько избыточных move-конструкторов.

## Время жизни во время цикла

В цикл передаётся (по rvalue-ссылке) not_matched_yet.

Внутри выполняется (или не выполняется) преобразование, и мы получаем
- ссылку на исходный not_matched_yet - превращаем в значение matched_final_halted
- значение matched_regular - превращаем в значение not_matched_yet
- значение matched_final

При разматывании цикла (`FWD(nmy) >> body >> body >> ... >> rule_loop{}`)
(где body инкапсулирует выполнение одной итерации с последующей заменой типа)
результат выражения может оказаться
- ссылкой на промежуточный `matched_final` или `matched_final_halted` (из `>> body`)
- окончательным значением (из рекурсии `>> rule_loop{}`)
- новым значением `not_matched_yet` (из рекурсии), если достигнута предельная длительность цикла

Поэтому, в отличие от забега по правилам, нужно всегда возвращать значение.

Таким образом, `rule_loop` выпадает из точного концепта `Rule`,
поскольку не гарантирует, что если на входе `not_matched_yet&&` и на выходе `not_matched_yet`,
то это тот же самый объект.

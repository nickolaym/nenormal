# Вычисления в циклах и композиция функций

У нас есть два вида циклов - цепочка ветвлений и бесконечный цикл.

Цепочка образована правилами подпрограммы.
```cpp
// псевдокод
template<Rule auto r1, Rule auto r2, ....., Rule auto rn> struct rules {
    auto operator()(CtStr auto text) const {
        for (auto r : {r1, r2, ..., rn}) {
            auto tr = r(t);
            if constexpr (!failed(tr)) return tr;
        }
        return failed{};
    }
};

Цикл - единственным правилом (подпрограммой верхнего уровня)
```cpp
// псевдокод
template<Rule auto r> struct rule_loop {
    auto operator()(CtStr auto text) const {
        auto tr = r(t);
        while (true) {
            if constexpr (failed(tr)) return r; // предыдущий результат
            r = (*this)(tr);
        }
    }
}
```
(для НАМ с правилами остановки будет чуть сложнее).

Но циклы в сишном понимании нам недоступны.

Хотя правила и принадлежат к одному концепту Rule, но даже не к одному шаблону rule<...>.
А строки - все разной длины.

Классический способ for-цикла - это рекурсия. Например, вот такая:
```cpp
template<Rule auto r1, Rule auto ... rs>
auto for_loop(auto t) {
    auto tr = r1(t);
    if constexpr(!failed(tr)) return tr;
    else return for_loop<rs...>(t);
}
```

У неё есть две тяжёлые цены.

1) Воплощается O(n) шаблонов `for_loop<r1,...,rn>`, `for_loop<r2,...,rn>`, ...
   общее количество аргументов которых - O(n^2).
2) Глубина рекурсии также становится O(n),
   при том, что компилятор накладывает ограничения (до 900).

Однако существует альтернатива - выражения свёртки.

# Выражение свёртки

```cpp
auto left_fold(auto arg, auto ... fs) {
    return (arg >> ... >> fs);
    // (((arg >> f1) >> f2) >> ... ) >> fn)
}
```
Чтобы оно сработало, необходимы следующие условия
- типы аргументов и результатов должны быть однородными, Arg | Stop
- операции `(>>f)` также должны быть однородными, Arg -> Arg | Stop
  (из более узкой категории в более широкую)
- преждевременный выход из цикла - это когда промежуточный результат попадает в Stop.

Для удобства мы максимально упростим вид оператора `>>`.
Пусть это будет обычное применение функции к аргументу, но с небольшой подсказкой:
- `stop{x} >> f = stop{x}`
- `arg{x}  >> f = f(arg{x})`

То есть, функции у нас тоже эндоморфизмы в категории ArgStop.

Зачем так сделано. Как только мы получили stop, то компилятор перестаёт просовывать
значение внутрь следующих функций, инстанцируя их шаблоны на всю ненужную нам глубину,
а сразу пробегает по оставшимся операторам.

А ещё - мы не задумываемся о содержании объектов arg и stop.

## Как сделать подпрограмму на свёртке?

В нашей задаче уже известно, что правила - CtStr -> CtStr|fail.

Казалось бы, просто давайте введём `operator >>` между строками и правилами!

Вот только результат CtStr - в категории Stop, а результат fail - **ни туда ни сюда**.

Поэтому нужно отобразить нашу категорию Rule `({CtStr, fail}, ->)`
в категорию ArgStop `({arg, stop}, {fun}, >>)` нетривиальным способом.

- `make_arg(s) = arg{s}` - действительно, аргументы у нас всегда - строки (пока что тривиально)
- `take_res(arg{_}) = fail{}` - если мы пробежали всю цепочку, но так и не получили результат, это провал
- `make_fun(rule)` - распаковывает строку из аргумента (arg{s}), находит `r=rule(s)`, и если это fail, то возвращает `arg{s}`, а если другая строка - то `stop{r}`

## Как сделать while-цикл на свёртке?

Ну уж казалось бы, цикл - это неизбежная рекурсия.

```cpp
auto while_loop(auto a, auto f) {
    auto r = f(a);
    if constexpr(failed(r)) return a;
    else return while_loop(r, f);
}
```

Да, но это не концевая, а обычная рекурсия (с точки зрения компилятора),
и её глубина равна O(n), где n - время работы программы.

А мы помним про ограничения...

Но заметим интересное свойство. Если мы перейдём в категорию ArgStop, то цикл станет вот таким
```cpp
auto while_loop(fun) {
    return [=](auto arg) { return arg >> fun >> while_loop(fun); };
}

// ну или чтобы не перевычислять лямбду каждый раз, то вот так
template<auto fun> struct while_loop {
    static constexpr myself = while_loop{};
    auto operator()(auto arg) const { return arg >> fun >> myself; }
}
```

Мы всё ещё имеем дело с рекурсией линейной длины-глубины, но уже избавились от явных ветвлений.

А ещё мы можем заметить ассоциативность применения функций.

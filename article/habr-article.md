# Ненормальное Марковское Программирование

Какие программы могут быть по-настоящему достойны хаба "**ненормальное** программирование"?

Конечно же, программы для **нормальных** марковских алгорифмов!

Но, будем честны перед собой: абстрактные машины - очень просты в реализации.
Зачастую вызов состоит в том, чтобы сделать какую-нибудь эзотерическую версию,
язык для машины на минималках, и затем ещё и минимизировать транслятор для него.

Но в рантайме-то любой дурак сделает. (см. [простенькие варианты реализации](easy-runtime-impl.md))

Поэтому поставим задачу со звёздочкой: затащить нормальные алгорифмы в компайл-тайм!

## Кратенько, что такое Нормальные Алгорифмы Маркова (НАМ)

Машина устроена следующим образом.

- Программа - это последовательность правил подстановки - троек
  - строка поиска
  - строка замены
  - признак "обычное/финальное правило" (то есть: продолжать/остановиться)
- Данные - это строка, над которой выполняется программа

За один такт машина
- ищет первое подходящее правило (искомая строка входит в строку данных),
- выполняет замену подстроки,
- и если правило финальное - завершает работу, а если обычное - уходит на следующий такт

Например, программа `[("world", "mir"), ("hello", "privet")]` над текстом `"hello world"` за первый такт заменит world на mir, несмотря на то, что hello там тоже есть.

Машина останавливается,
- когда было применено финальное правило
- когда ничего не удалось найти (и заменить).

## Строительные кубики

### Строки

Прежде всего, нам нужны строки во время компиляции. Причём человекочитаемые, а не старое-недоброе вида
```cpp
template<char...> struct s_t_r_i_n_g {};
constexpr auto hello = s_t_r_i_n_g<'h', 'e', 'l', 'l', 'o'>
```
(да, я сознательно выразил своё отношение к этому подходу через имя этого шаблона!)

К счастью, это не так уж сложно.

Хотя строковые литералы не являются первоклассными сущностями времени компиляции, но массивы символов - являются. Нужно лишь завернуть их в структуру:
```cpp
template<size_t N> struct str { char data[N]; };

constexpr auto hello = str{"hello"}; // будет выведен N=6
```
Ой! Строка длиной 5 символов имеет размер 6? Это так по-сишному, но как-то не по-сиплюсплюсному...

Но мы учтём это: договоримся, что такие структуры всегда будем конструировать из литералов с неявным концевым нулём. И будем возвращать size() = N-1.

Заодно нахаляву получили приведение нашей компайл-тайм-строки к сишной, что позволит выводить её с помощью printf, например!

### Зависимые типы

Давайте посмотрим на функцию поиска-и-замены. Её рантаймовая сигнатура выглядит примерно так
```cpp
std::optional<std::string> try_substitute(std::string src, std::string const& search, std::string const& replace);
```

Во время компиляции - допустим, мы знаем размеры строк-аргументов (конечно же, знаем!), но какой будет тип результата?
Допустим, пара
- строка длины `max(0, src.size()-search.size()+replace.size()))`
- булев флажок "не смогли / смогли"

Или же мы попробуем возвращать разные типы
- "не смогли"
- результирующая строка, если смогли

с тем, чтобы дальше делать ветвление по типам... и это избавило бы нас от распухания буфера под максимально возможную строку.

Но вот проблема! Тип результата (и вся сигнатура функции) может зависеть только от типов аргументов, но не от их значений.

Даже если мы зафиксировали содержимое search и replace извне (как параметры шаблона правила подстановки),
то зафиксировать содержимое входной строки мы не можем.

Но выход есть! Это типы, параметризованные значениями!
```cpp
template<auto V> struct ct {
    static constexpr auto value = V;
};
```

Поскольку экземпляры типа `str<N>` могут быть константами времени компиляции,
то их можно использовать как параметры шаблона `ct`.

Теперь наша функция приобретает вид
```cpp
auto /* nothing | just<ct<str<N>{...}>> */
/* выведем тип из аргументов... внутри, на if-constexpr */
try_substitute(
    auto /*ct<str<N1>{...}>*/ src,
    auto /*ct<str<N2>{...}>*/ search,
    auto /*ct<str<N3>{...}>*/ replace
) {
    if constexpr (/* не смогли подставить */)
        return nothing{};
    else if constexpr (/* подстановка тривиальная */)
        return just{src};
    else
        constexpr auto dst = /* сделали подстановку */;
        return just{ct<dst>{}};
}
```
Почему не голая `ct<str>`, а ещё и обёртка в just? Потому что в дальнейшем удобно ограниченно припахать монаду Maybe.

Итак, мы избавились от главной головной боли - как сделать переменные строки, не резервируя под них всё больший буфер.

### Циклы

И тут нас ждёт следующая засада. Нам каким-то способом надо сделать два цикла - for по списку правил и while по тактам программы.

### Цикл for

Да, constexpr-функции позволяют выполнять циклы, но... над элементами одного типа.
А у нас все правила параметризованы разными строками. Разной длины и разного содержимого.

Конечно, можно попробовать исхитриться: завести гомогенный массив `const char*`, найти совпадение, узнать индекс, и по индексу извлечь правило из кортежа (гетерогенного массива) и уже затем выполнить замену, получив новую строку нового уникального типа.

Или же сделать рекурсию по-старинке:
- взять гетерогенный список правил (variadic template),
- попробовать применить первое правило из него,
- если не получилось, то рекурсивно продолжить с хвостом этого списка
```cpp
auto single_iteration(auto src, auto p1, auto... ps) {
    auto maybe_dst = try_substitute(src, p1);
    if constexpr (/* смогли */)
        return /* выковыряли результат из maybe_dst */;
    else
        return single_iteration(src, ps...);
}
```
Вот только правил у нас в НАМ-программе может быть очень много.

И мы замучаем компилятор и количеством аргументов, и глубиной рекурсии, и количеством воплощений шаблонов функции забега.

Хорошо бы как-то развернуть такой цикл внутри функции...
```cpp
auto single_iteration(auto src, auto p1, auto p2, auto p3, ..., auto pn) {
    { auto x1 = try_substitute(src, p1); if constexpr (x1 успешно) return x1; else
    { auto x2 = try_substitute(src, p2); if constexpr (x2 успешно) return x1; else
    { auto x2 = try_substitute(src, p3); if constexpr (x3 успешно) return x1; else
    ...
    { auto xn = try_substitute(src, pn); if constexpr (xn успешно) return xn; else
    { return неуспешно(src);
    }} ... }}}
}
```
вот только variadic template на такие сложносоставные стейтменты не распространяются.

Но выход есть! :) Это fold-expressions.

И нам придётся реализовать монаду Either.

`(Right src) >>= p` - вернёт `(Left dst)` в случае успешной подстановки, и `(Right src)` в случае неуспешной.

Тогда `(Right src) >>= p1 >>= p2 >>= ... >>= pn` после успешной подстановки проигнорирует весь хвост выражения.

И мы можем обеспечить хаскельную ленивость средствами типизации C++.

Просто напишем перегрузки оператора `>>` такие, что для некоего стартового типа-обёртки `not_matched_yet` он вызывает функцию подстановки, а для некоего `matched` - сразу возвращает.

```cpp
template<class T> struct not_matched_yet {
    T value;
    constexpr auto operator >> (auto p) const { return p(value); }
    // где p возвращает matched / not_matched_yet
};
template<class T> struct matched {
    T value;
    constexpr auto const& operator >> (auto p) const { return *this; }
};
```
Осталось выполнить эту свёртку
```cpp
auto /* matched|not_matched_yet */ run_iteration(auto src, auto... ps) {
    return (not_matched_yet{src} >> ... >> ps);
}
```

Ну а запихать `ct<str>` в обёртку и извлечь обратно - задача тривиальная.

Позже я скажу, что не всё тут так просто, но... в первом приближении мы смогли.

### Цикл while

Хорошо, в for мы хоть и на стадии инстанцирования, но знаем фиксированное количество элементов.

А с циклом while как быть, ведь проблема останова никуда не делась?

К сожалению, тут выход один - рекурсия. И к сожалению, концевой рекурсии в C++ нет.

(Если есть, то расскажите, буду премного благодарен)

```cpp
auto run(auto src, auto program) {
    auto tmp = single_iteration(src, program);
    if (tmp это not_matched_yet или matched final)
        return tmp.value;
    else
        return run(tmp.value, program);
}
```

И у компилятора глубина рекурсии - это ограниченный ресурс. Скажем, у gcc по умолчанию она равна 900.

Но если мы снова взглянем на Either и воспользуемся правилом ассоциативности,
```haskell
run_while program x = Right x >>= program >>= (run_while program)

= Right x >>= program >>= (program >>= run_while program)
= Right x >>= program >>= program >>= program >>= ... >>= (run_while program)
```

И просто развернув цикл в 10 итераций, мы увеличим максимальную глубину до 9000.

А если сделаем адаптер, который будет умножать применяемую функцию в N раз, то вообще получим N^900 или что-то в таком роде.
```cpp
template<class P> struct n_times {
    P p;
    constexpr auto operator()(auto x) const {
        return not_matched_yet{x} >> p >> p >> p >> p >> p; // n раз, сколько нам не жалко
    }
};

template<class P> struct linear_loop {
    P p;
    constexpr auto operator()(auto x) const {
        return not_matched_yet{x} >> n_times{p} >> *this;
    }
};

template<class P> struct growing_loop {
    P p;
    constexpr auto operator()(auto x) const {
        return not_matched_yet{x} >> n_times{p} >> growing_loop{n_times{p}};
    }
};
```

Только телом цикла должна быть функция со следующим поведением:
- если ни одно правило не выполнилось, то возвращается Left
- если выполнилось финальное правило, то опять же возвращается Left
- если выполнилось обычное правило, то возвращается Right, чтобы мы продолжили работу

### Наивный тип для правила

Отсюда следует, что каждое правило НАМ-программы - это функция вида
```haskell
type Rule = String -> Either String (Either String String)

not_matched_yet x = Right x
matched_regular x = Left (Right x) -- вышли из for, но остались в while
matched_final   x = Left (Left x) -- вышли из for и из while
```

И тело цикладолжна будет превратить результат цепочки правил
- `Left xz` в xz (где xz - это Left или Right в зависимости от вида сработавшего правила)
- `Right x` - в `Left x`

Ну, или в конец программы нужно дописать неявное финальное правило `"" ->. ""` -
которое гарантированно остановит цикл, если никакие другие правила не подошли.

Тогда результат цепочки - это всегда `Left xz`, и не потребуется ветвления.

### Но компилятор сказал "где мой swap-файл?"

Опыты показали, что вот такая конструкция Either-Either создаёт очень большую нагрузку на компилятор.

Маленькие НАМ-программы и короткие запуски он съедает, а на длинных кончается оперативная память.

Поэтому практичный подход - зафиксировать, что у нас есть 3 (или 4) типа
- not_matched_yet
  - вызовет правило
  - в конце тела цикла превратится в matched_final (или matched_final_halted)
- matched_regular - в конце тела цикла превратится в not_matched_yet
- matched_final - останется самим собой
- matched_final_halted - останется самим собой

То есть, у каждого из этих типов есть методы
- `operator >>(auto p)`
- `commit_loop()`

и компилятор очень быстро и достаточно лениво развернёт всю последовательность операций.

А главное, теперь у нас вместо неочевидных `Left (Right x)` - внятно именованные типы.

Монада Tristate - это тот же Either, только узкоспециализированный.

## Итого

У нас получились следующие детальки
```cpp
template<auto search, auto replace, auto kind> struct rule {
    constexpr auto operator()(auto src) const {
        // возвращает одно из трёх
        // - not_matched_yet{src}
        // - matched_regular{dst}
        // - matched_final{dst}
    }
}

template<auto... ps> struct rules {
    constexpr auto operator()(auto src) const {
        return (not_matched_yet{src} >> ... >> ps);
    }
};

template<auto program> struct loop_body {
    constexpr auto operator()(auto src) const {
        return (not_matched_yet{src} >> program).commit_loop();
    }
};

template<auto body> struct loop_impl {
    constexpr auto operator()(auto src) const {
        return (not_matched_yet{src} >> body >> loop_impl<n_times<body>>;
    }
};

// и наконец, вне этой иерархии rule-подобных типов,
template<auto program> struct machine {
    constexpr auto operator()(auto src) const {
        constexpr auto body = loop_body<program>{};
        constexpr auto loop = loop_impl<body>{};
        return (not_matched_yet{src} >> loop).value;
    }
};
```
Причём обратите внимание, все эти типы - зависимые типы, подобные ct.
Поэтому они передаются друг в друга через value-параметры шаблонов.

Можно было бы сделать их классическими обёртками `template<class T> struct some{T value}`, но
- во-первых, не нужно, потому что элементарные правила - всё равно зависимые типы
- во-вторых, rules с переменным числом полей - нельзя просто так сделать,
  надо внутри завести поле-кортеж... которое будет содержать пустые структуры
  (либо структуры пустых структур)

## Давайте уже что-нибудь напишем на НАМ?

А уже потом вернёмся к кое-каким довольно болезненным техническим деталям!

### hello world

Итак, самое простое - это проверить, что НАМ работает.

Программа из единственного правила...
```cpp
constexpr auto helloworld = MACHINE(FINAL_RULE("", "hello, world!"));

constexpr auto src = CTSTR("");
constexpr auto dst = helloworld(src)
static_assert(dst == CTSTR("hello, world!")); // проверили во время компиляции!
```
Ожидаемо.

Ну и чуть посложнее
```cpp
constexpr auto privetmir = MACHINE(RULES(
    RULE("hello", "privet"),
    RULE("world", "mir")
));

constexpr auto src = CTSTR("hello, world!");
constexpr auto dst = privetmir(src);

static_assert(dst == CTSTR("privet, mir!")); // проверили во время компиляции!
```
Тут ничего необычного, два независимых правила, порядок их срабатывания значения не имеет.

### Порядок имеет значение

Давайте что-нибудь, где порядок имеет значение.
```cpp
constexpr auto abcdef = MACHINE(RULES(
    RULE("a", "b"),
    FINAL_RULE("c", "d"),
    RULE("e", "f")
));

static_assert(abcdef(CTSTR("aaa")) == CTSTR("bbb"));
static_assert(abcdef(CTSTR("ccc")) == CTSTR("dcc")); // остановились после первой же "c"
static_assert(abcdef(CTSTR("eee")) == CTSTR("fff"));

static_assert(abcdef(CTSTR("ace")) == CTSTR("bde")); // до "e" дело не дошло, логично
static_assert(abcdef(CTSTR("eca")) == CTSTR("edb")); // немного неочевидно, ведь e перед c?
```

Тут важный момент - в том, что машина ищет не самое левое вхождение из всех искомых строк, а самую первую искомую строку из набора. И неважно, что в тексте эта подстрока оказалась правее других.

Это даёт нам подсказку, как программировать на НАМ.

### Правильная скобочная последовательность

Пусть на входе строка, содержащая скобки в произвольном порядке (и ничего, кроме скобок).

Если они сбалансированы, то надо вывести "GOOD", а если нет - то "BAD".

Алгоритм работы:
- сперва сократим все парные скобки: `"()" -> ""`
- затем приведём все непарные к одному виду:
  - `"(" -> "_"`
  - `")" -> "_"`
- затем сократим всю серию непарных:
  - `"__" -> "_"`
- затем, если осталась непарная (единственная) - выведем и остановимся
  - `"_" ->. "BAD"`
- наконец, если ничего не осталось - выведем и остановимся
  - `"" ->. "GOOD"`

```cpp
auto cps = MACHINE(RULES(
    RULE("()", ""),
    RULE("(", "_"), RULE(")", "_"),
    RULE("__", "_"),
    FINAL_RULE("_", "BAD"),
    FINAL_RULE("", "GOOD")
));

static_assert(cps(CTSTR("")) == CTSTR("GOOD"));
static_assert(cps(CTSTR("((()))()()")) == CTSTR("GOOD"));
static_assert(cps(CTSTR("((())))()())(()")) == CTSTR("GOOD"));
```

Здесь правила идут в том же порядке, что и наш план работы.

По сути, программа распалась на последовательность циклов:
- пока есть пары, срабатывает первое правило, дальше не идём
- пока есть скобки (теперь только непарные), первое правило не срабатывает, но срабатывает второе
- пока есть несколько подчёркиваний (а скобок больше не осталось), срабатывает третье
- ну и два финала

Мы не можем перемешать правила, но и не хотим.

### А что там с проблемой останова?

Ведь НАМ эквивалентны машине Тьюринга.

А на чём бы её проверить? Конечно же, на всеми любимой гипотезе Коллаца!

Сложные реализации смотрите на гитхабе, а здесь я приведу упрощённую версию.

Пусть на входе строка из N единичек. (То есть, будем в единеричной системе счисления работать).

- Если N=1, остановимся.
- Если N чётно, то сократим их вдвое
- Если нечётно - утроим и прибавим единицу.

Последовательный алгоритм выглядит так:

1. Если в строке ровно одна единичка, остановимся `"1" ->. "STOP"`
2. Проверим чётность (а заодно и поделим пополам)
  - добавим в начало строки маркер деления `"" -> "[div]"`
  - и побежим вправо `"[div]11" -> "1[div]"`
3. Когда маркер добежал до конца строки, там либо ничего не осталось, либо осталась одна единичка.
  - если ничего, то просто удалим маркер, `"[div]" -> ""` и вернёмся к началу
  - если единичка, то перейдём к шагу 3 - выполним N*3+1
4. Выполним утроение, с учётом того, что слева (N-1)/2 единичек, а справа 1
  - учтём, что `N*3+1` = `(N-1)/2*6 + 4`
  - запустим новый маркер `"[div]1" -> "[mul]1111"`
  - и побежим влево `"1[mul]" -> "[mul]111111"`
5. Когда маркер добежал до начала строки, просто удалим его `"[mul]" -> ""` и вернёмся к началу

Итак, у нас есть следующие правила
```cpp
constexpr auto stop   = FINAL_RULE("1",       "");
constexpr auto div_start    = RULE("",        "[div]");
constexpr auto div_continue = RULE("[div]11", "1[div]");
constexpr auto div_end      = RULE("[div]",   "");
constexpr auto mul_start    = RULE("[div]1",  "[mul]1111");
constexpr auto mul_continue = RULE("1[mul]",  "[mul]111111");
constexpr auto mul_end      = RULE("[mul]",   "");
```
но очевидно же, что если собрать их в таком порядке, то мы мгновенно остановимся и получим строку
вида `"STOP111111"одинодинодин!`

Поэтому stop должно быть в самом конце.

Тогда очевидно, что div_start нельзя делать первым, иначе получим взрыв `"[div][div][div].....111"`

И так далее. Фактически, нам нужно перевернуть весь алгоритм задом наперёд.

Но не совсем! Например, если мы поместим mul_end перед mul_continue,
то просто удалим маркер умножения, не добежав до левого края.

По сути, нам надо упорядочить правила так, чтобы искомые НАД-строки предшествовали ПОД-строкам.

Делаем вторую попытку...
```cpp
constexpr auto mul_continue = RULE("1[mul]",  "[mul]111111");
constexpr auto mul_end      = RULE("[mul]",   "");
constexpr auto div_continue = RULE("[div]11", "1[div]");
constexpr auto mul_start    = RULE("[div]1",  "[mul]1111");
constexpr auto div_end      = RULE("[div]",   "");
constexpr auto div_start    = RULE("",        "[div]");
constexpr auto stop   = FINAL_RULE("1",       "STOP");
```
Если поместим div_start перед stop, то никогда не достигнем финала,
так и будем прыгать между div_start и div_end.

Если поместим stop перед div_start, то получим старое доброе `"STOP111111"одинодинодин!`.

Решение - запускать деление не с 0, а с 2.
```cpp
// сами правила мы можем написать в каком угодно порядке
constexpr auto mul_continue = RULE("1[mul]",  "[mul]111111");
constexpr auto mul_end      = RULE("[mul]",   "");
constexpr auto div_continue = RULE("[div]11", "1[div]");
constexpr auto mul_start    = RULE("[div]1",  "[mul]1111");
constexpr auto div_end      = RULE("[div]",   "");
constexpr auto div_start    = RULE("11",      "1[div]");
constexpr auto stop   = FINAL_RULE("1",       "STOP");
constexpr auto zero   = FINAL_RULE("",        "ZERO"); // а вот нечего, ибо!

// но в программе они должны идти в строго заданном
constexpr auto kollatz = MACHINE(RULES(
    mul_continue,
    mul_end,
    div_continue,
    mul_start,
    div_end,
    div_start,
    stop,
    zero
));
```

### Как ещё можно программировать?

Определённую головную боль доставляет распознавание начала и конца строки.

У НАМ нет метасимволов `^` и `$`, но программисту подконтрольна спецификация на формат входных данных.

Для каких-либо наших нужд мы можем потребовать, чтобы текст был обрамлён этими символами.

Тот же мини-Коллац: если строка обрамлена, то мы можем вместо хитрого порядка правил
ввести уникальные тэги для каждого состояния алгоритма.

Тогда в каждый момент времени подойдёт ровно одно правило, а значит, их порядок неважен!

```
^$          ->. ZERO
^1$         ->. STOP

^11         -> ^[run][*2+]11 -- изолировали маркер начала и запустили цикл деления
[*2+]11     -> 1[*2+]        -- цикл деления (слева направо)

[*2+]$      -> [+]$          -- закончили деление: чётное - запустили цикл сложения
1[+]        -> [+]1          -- цикл сложения (справа налево)
^[run][+]   -> ^             -- закончили сложение и убрали изолятор

[*2+]1$     -> [*6+]1111$    -- закончили деление: нечётное - запустили цикл умножения
1[*6+]      -> [*6+]111111   -- цикл умножения (справа налево)
^[run][*6+] -> ^             -- закончили умножение и убрали изолятор
```

### Десятичное сложение

## Аугментация

Чистая функция времени компиляции - на входе строка, на выходе строка -
это просто такой способ заставить компилятор (возможно, очень долго) вычислить константу.

Но чтобы посмотреть на промежуточные шаги, мы прибегнем к монаде Writer.

Вместо `Tristate Str` запустим `Tristate (Writer Str)`.

И каждый раз, когда элементарное правило выполняет подстановку,
оно заодно дёргает колбек, который обновляет значение аугментации.

Естественно, универсальный Writer нам не нужен, сделаем специально под задачу.

Три основных сценария
- ничего не делать, - колбек вырожденный, состояние вырожденное
- выполнять побочный эффект (при этом машина может пересть быть constexpr, но по-прежнему выводит тип-и-значение во время компиляции)
- обновить состояние: `a' = f(a, ...)` (при этом если колбек - чистая constexpr-функция, то и машина - чистая)

Помимо прочего, у нас открываются новые возможности: хакнуть программу.

- обёртка hidden_rule - делает так, что замена текста производится, но колбек не применяется
- обёртка facade_rule - делает так, что если замена текста произошла, то колбек применяется не к элементарному правилу где-то внутри, а непосредственно к фасаду

Для этого они расшивают `Writer Str` до - отдельно Writer, отдельно Str (ну или заменяют Writer на вырожденный), передают управление внутрь, а потом сшивают их обратно.

```cpp
template<auto s, auto r, auto k> struct rule {
    constexpr auto operator()(auto input) const { // Augmented -> Tristate Augmented
        constexpr auto input_text = input.text;
        constexpr auto result_text = this->do_it(input_text); // Tristate Str
        if constexpr (result_text is not_matched_yet) {
            return not_matched_yet{input};
        } else {
            return result_text.replace_value(
                augmented_text{
                    result_text.value,
                    // вызываем колбек - обновляем Writer
                    input.aux(input_text, *this, result_text.value)
                }
            );
        }
    }
};

template<auto p> struct hidden_rule {
    constexpr auto operator()(auto input) const { // Augmented -> Tristate Augmented
        constexpr auto bare_input = augmented_text{input.text, empty{}};
        constexpr auto bare_output = p(bare_input); // Tristate
        return bare_output.replace_value(
            augmented_text{bare_output.value.text, input.aux}
        );
    }
};

template<auto p> struct facade_rule {
    constexpr auto operator()(auto input) const { // Augmented -> Tristate Augmented
        constexpr auto bare_input = augmented_text{input.text, empty{}};
        constexpr auto bare_output = p(bare_input); // Tristate Augmented Empty
        if constexpr (bare_output is not_matched_yet) {
            // как hidden
            return bare_output.replace_value(
                augmented_text{bare_output.value.text, input.aux}
            );
        } else {
            return bare_output.replace_value(
                augmented_text{
                    bare_output.value.text,
                    // вызываем колбек - обновляем Writer
                    input.aux(input.text, *this, bare_output.value.text)
                }
            );

        }
    }
};
```

### Что тут не очень хорошо?

Это всё хорошо, но оказалось, что мы заставляем компилятор делать кучу лишней работы.

### От однородной монады Writer - к смеси Writer | Id

Во-первых, если `hidden_rule` или `facade_rule`, - или даже пользователь с самого начала, -
передаёт внутрь вырожденный колбек `empty{}`, - то компилятор, а затем и рантайм,
создаёт все эти структуры с вырожденными данными, порождая гору ненужных типов,
применяет вырожденные колбеки, выводя тип результата, - и тратит довольно много сил.

Особенно, если нам в конкретном случае вовсе не нужна была аугментация.

Выход очень простой: сделать перегрузку оператора (или его внутренностей),
чтобы тот одинаково работал как с `Writer Str`, так и с `Str` (по сути, `Id Str`)

Для этого нам нужны две перегруженные функции:
- извлечь текст из монады (`Id` или `Writer`)
- запихать в монаду новый текст и все данные для вызова колбека

```cpp
CtStr auto extract_text(CtStr auto t) { return t; }
CtStr auto extract_text(Augmented auto t) { return t.text; }

CtStr auto update_text(CtStr auto old, CtStr auto new_text, auto... args) {
    return new_text;
}
Augmented auto update_text(Augmented auto old, CtStr auto new_text, auto... args) {
    return augmented_text{new_text, old.aux(old.text, args..., new_text)};
}
```

### От тотальных функций - к частным

Во-вторых, если правила (и их наборы и обёртки) - это тотальные функции `Something -> Tristate Something`
то при их вызове мы каждый раз расшиваем `not_matched_yet`, извлекая оттуда пусть даже ссылку на значение,
а при выходе из них - при неудаче, - вынужденно создаём новую копию `not_matched_yet`.

Даже если переписать всё с move-семантикой, мы делаем столько конструкторов,
сколько у нас есть элементарных правил, наборов правил, обёрток над наборами,
плюс тело цикла, плюс цикл целиком, - помножить на количество тактов.

Это очень, очень много.

А вот если перейдём к частичным функциям, - принимающим и возвращающим Tristate,
но при этом поклянёмся, что на вход приходят только `not_matched_yet`, -
то сможем, аккуратно возвращая ссылки на аргумент, полученный по ссылке, резко сократить расход памяти.

(А память-то в компайл-тайме очень дорогая).

Теперь `rules<ps...>` выглядит вот так
```cpp
template<auto... ps> struct rules {
    // было
    constexpr auto operator()(auto input) const {
        return not_matched_yet{std::move(input)} >> ... >> ps;
    }
    // стало
    constexpr decltype(auto) operator()(auto&& nmy) const {
        return (nmy >> ... >> ps).commit_alts();
    }
};
```

### От возврату по значению - к тонкому управлению ссылками

Что это за `commit_alts`, зачем он?!

А дело вот в чём.

На входе у нас nmy (ссылка на `not_matched_yet`).

- Если по результатам забега мы получили ссылку на `not_matched_yet`, то мы клянёмся, что это та же самая ссылка.
  Возвращаем эту ссылку.
- Если получили значение (например, `matched_regular`) - то `decltype(auto)` будет rvalue `matched_regular`.
  Возвращаем значение, тут всё ок.
- Если же внутри забега получили значение `matched_regular`, то его `operator >>` вернёт rvalue-refarence, то есть, ссылку... и `decltype(auto)` тоже будет ссылкой.
  Ссылкой на локальный временный объект, который сдохнет после выхода из полного выражения!

Вот чтобы такого не было, мы для каждого типа из Tristate определим разное поведение -
каким способом завершать забег `>> ... >>`
- `not_matched_yet` - возвращает себя по ссылке
- все остальные - возвращают себя по значению

```cpp
template<class T> struct not_matched_yet {
    constexpr decltype(auto) commit_alts() const& { return *this; }
    constexpr decltype(auto) commit_alts() && { return *this; }
};

template<class T> struct matched_regular {
    constexpr auto commit_alts() const& { return *this; }
    constexpr auto commit_alts() && { return *this; }
};
```

## Технические детали: концепты

В проекте вовсю используются концепты. Причин для этого несколько:

- Попросту для улучшения читаемости.
  Программирование на шаблонах, где много `<class T>` и `auto` слишком сильно стирает подробности,
  и нужно давать `long_mnemonic_identifiers`, чтобы подсказать читателю, что же стоит за каждым
  параметром или аргументом.
- Для ошибкобезопасности. Если подсунуть неправильный тип, то компилятор может споткнуться
  или сразу же в точке вызова, или очень глубоко внутри, - сформировав длиннющую и невменяеемую ошибку
- Для разрешения перегрузок. По меньшей мере, приходится отличать голые строки от аугментированных.

Можно ли по-старинке вместо `void f(Some auto x, Another auto y)`
писать `template<.....> void f(some<...> x, another<...> y)` ?

Можно, но!
- это очень громоздко
- особенно, когда типы аргументов параметризованы значениями, а не типами
- для шаблонов типов, параметризованных шаблонными значениями, невозможно (попросту некуда) вынести
  именованные параметры шаблонов их типов.

```cpp
// C++20, concepts
template<Some auto v> struct another {.....};

// C++17
template<class T, some<T> v> struct another {.....};

constexpr some<U> s{123};
constexpr another< U, s > a;  // придётся указать никому не нужную зависимосить - тип U
```

Кроме прочего, концепты позволяют объединять несколько типов в один тайпкласс.
Конечно, не так строго, как в хаскелле, но лучше так, чем никак.

### Ограничение концептов

Концепт может быть параметризован.

Например, есть универсальный зависимый тип ct (compile-time)
```cpp
template<auto V> struct ct {
    static constexpr auto value = V;
    using type = std::remove_cv_t<decltype(V)>;
};

// какой угодно ct
template<class T> concept Ct = /* проверить, что T = ct<???> */;

// ct конкнетного типа
template<class T, class VT> concept CtOf = Ct<T> && std::same_as<typename T::type, VT>;
```
но, к сожалению, параметром концепта может быть только тип.

Сделать концепт, параметризованный концептом, нельзя.

Придётся руками расписывать
```cpp
template<size_t N> struct str { ..... };

template<class T> concept Str = /* T = str<N> */;

template<class T> concept CtStr = Ct<T> && Str<typename T::type>;
// вместо того, чтобы было
template<class T> concept CtStr = CtOfConcept<T, Str>;
```

Конечно, можно было бы выкрутиться, заведя параллельно каждому концепту метафункцию
```cpp
template<class T> struct is_str_t : std::false_type {};
template<size_t N> struct is_str_t<str<N>> : std::true_type {};
template<class T> concept Str = is_str_t<T>::value;

template<class T> struct is_ct_t : std::false_type {};
template<auto V> struct is_ct_t<ct<V>> : std::true_type {};
template<class T> concept Ct = is_ct_t<T>::value;

template<class T, class P> concept CtOfPredicate = Ct<T> && P<typename T::type>::value;
template<class T, class P> using is_ct_of_predicate_t = std::bool_constant<CtOfPredicate<T, P>>;

template<class T> concept CtStr = CtOfPredicate<T, is_str_t>;
template<class T> using is_ct_str = std::bool_constant<CtStr<T>>;
```

Поскольку структур и концептов в проекте много, то я предпочёл вариант с минимумом писанины.

```cpp
#define CONCEPT(name) template<class T> concept name = T::this_is_##name;

#define REPRESENTS(name) static constexpr bool this_is_##name = true;

CONCEPT(Foo);

struct foo1 {
    REPRESENTS(Foo);
};
struct foo2 {
    REPRESENTS(Foo);
};
```

Кроме прочего, этот способ проверки избавляет от проблем с правилами видимости и порядком объявлений.

Если бы, например, делать на специализации метафункции,
```cpp
template<class T> struct is_foo_t : std::false_type {};

template<class T> concept Foo = is_foo_t<T>::value;

template<class X, class Y> struct foo1 {};
template<class X, class Y> struct is_foo_t<foo1<X, Y>> : std::true_type {};
```
то объявление концпета и связанной метафункции должно строго предшествовать объявлению представителей.

Из-за чего не получится без выкрутасов применять конецепт самого себя внутри класса.
